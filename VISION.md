# VISION: Независимый клинико-экономический Decision Engine

## Питч (одной фразой)

Мы делаем независимый AI-движок для лабораторной диагностики, который оптимизирует
доборы анализов, снижает перерасход и повышает клиническую ценность диагностики —
через B2C-сервис, партнёрство с КДЛ, корпоративные программы и ДМС.

---

## 0. Большая идея

Не "предсказание болезней по анализам".

**Независимый клинико-экономический decision engine**, который:

- Анализирует рутинные лабораторные данные + демографию
- Не ставит диагнозы, а:
  - Делает триаж (кому и какие доборы реально нужны)
  - Оценивает интерпретируемость результатов
  - Оптимизирует расходы на диагностику
  - Снижает ошибки pre-analytical этапа
- Работает как:
  - **B2C-сервис** (самостоятельно, без зависимостей)
  - **Встраиваемый слой в КДЛ** (win-win: им выручка, нам данные)
  - **Надстройка для работодателей / ДМС** (этап 2)
  - **Модуль для МИС/ЛИС или слой для страховых** (масштаб)

### Ключевая ценность

Делать лабораторную диагностику дешевле, безопаснее и осмысленнее
для тех, кто платит за систему (пользователь, работодатель, страховая, клиника),
**не вступая в конфликт интересов с КДЛ**.

## 1. Ядро технологии (почему это "не калькулятор")

### Risk & Policy Engine

Не просто пороги (TSH<0.3), а выбор маршрута с учётом:
- Клинического риска (красные флаги из паттерна, не отдельных значений)
- Вероятности полезности добора (P из CatBoost, не из LLM)
- Вероятности комплаенса пациента
- Экономики канала (B2C vs ДМС vs корпоратив)
- Операционных ограничений (реагенты, очереди, сезонность)

### Pre-analytical Validator

Проверка корректности интерпретации результатов:
- Ферритин при воспалении → рекомендовать CRP/sTfR
- Бессмысленные комбинации анализов
- Неподходящие условия для интерпретации

> **Статус**: потенциальная фича. Нужен custdev — какой % оптимизации реально
> есть на рынке, неизвестно. Может стать бонусным модулем для КДЛ/страховых.

### Recommendation Engine (клинически обоснованный)

Кому предложить дообследование, когда, с каким обоснованием:
- Только при наличии клинической пользы (паттерн из модели, PPV >= 60%)
- Прозрачное обоснование через SHAP (пациенту и врачу видно "почему")
- Не каннибализирует существующие панели (контуры доступа)
- Формулировки: информационные, не директивные ("рекомендуем", не "вам нужно")

### Три режима (из литературы)

| Режим | Что делает | Источник |
|---|---|---|
| **Reflex** | CBC → "нужен ферритин, добери" | McDermott & Baron, MGH |
| **Low Info** | "Этот тест будет нормальным, не назначай" | Levick & Baron, MGH |
| **Predict** | "Вот вероятность дефицита железа без теста" | MGH ferritin prediction |

Все три = одна модель с разным порогом:
- P(ID) > 0.6 → reflex (добери)
- P(ID) < 0.1 → low info (не назначай)
- Между ними → зона врачебного решения

## 2. Продуктовые гипотезы (обзор)

| # | Гипотеза | Кому | Что | Этап |
|---|---|---|---|---|
| **F** | **B2C "Загрузи анализы"** | Пользователь | Расшифровка + рекомендация + экономия | **СЕЙЧАС** |
| **D** | **"Умный добор" для КДЛ** | КДЛ (win-win) | Доп. выручка с текущего потока | **СЕЙЧАС** |
| **A** | Биохим-аудит продуктивности | Работодатель / ДМС | ML-триаж → точечные доборы | Этап 2 |
| **E** | Корпоративный паспорт здоровья | Работодатель + сотрудник | Карточки риска + добор | Этап 2-3 |
| **B** | White-label / интеграция МИС | МИС-вендор / Инвитро | AI-модуль | Масштаб |
| **C** | Аудитор для страховых | Страховая / TPA | Проверка обоснованности | Масштаб |

> Подробно по каждой гипотезе — в `BUSINESS_MODELS.md`
> Флоу и экономика — в `FLOWS.md`

### Гипотеза F: B2C (почему это первый приоритет)

```
Пользователь загружает ОАК → получает:
  ┌─────────────────────────────────────────┐
  │  Gauge вероятности (P из CatBoost)      │  ← LLM не умеет
  │  SHAP: почему модель так решила         │  ← LLM не умеет
  │  Перцентили: "вы среди 82% женщин..."   │  ← ни у кого нет (NHANES)
  │  Светофор: не отдельные, а комбинации   │  ← LLM сравнивает по одному
  │  1 рекомендация + 1 кнопка              │
  │  1-2 совета по питанию                  │
  │  Дисклеймер                             │
  └─────────────────────────────────────────┘
```

Ключевое отличие от BloodGPT / Kantesti / GPT-ботов:

```
Они: LLM сравнивает каждое значение с референсом ПО ОТДЕЛЬНОСТИ
     MCV=79 → "чуть ниже нормы". HGB=122 → "нижняя граница".
     Итог: "в целом ок, контроль через 3-6 мес"
     → ПРОПУСК ДЕФИЦИТА

Мы:  CatBoost видит ПАТТЕРН из 14+ параметров ОДНОВРЕМЕННО
     MCV+MCH+RDW+HGB+PLT+пол+возраст → P(дефицит) = 0.73
     → РЕКОМЕНДАЦИЯ: ферритин
     → PPV 65-70%: подтверждается у большинства
```

### Гипотеза D: КДЛ "умный добор" (почему это второй приоритет)

```
Пациент сдаёт ОАК → наша модель → 25% в жёлтой зоне →
SMS от имени КДЛ: "рекомендуем ферритин, скидка 15%"

КДЛ:     +217k/мес доп. выручки (с одной точки)
Мы:      20-30% от маржи = ~38k/мес/точку
Пациент: нашёл реальную проблему за 425 руб вместо панели за 7k
Врач:    не трогаем вообще
```

### Гипотеза B: МИС / Инвитро (стратегия масштабирования)

```
СЕЙЧАС:   Выходов на МИС нет. Фокус на B2C + КДЛ напрямую.

ВАРИАНТ 1 (Инвитро / крупная КДЛ):
  Если внутренняя ИС подходит для интеграции — работаем через них.
  Плюсы: быстрый масштаб, готовая инфра
  Минусы: зависимость, они "продвиженцы решения"

ВАРИАНТ 2 (самостоятельность):
  IP-лицензия макс., заходим в КДЛ напрямую.
  Плюсы: свой бренд, свобода, больше маржи
  Минусы: медленнее масштаб

ВАРИАНТ 3 (МИС-модуль):
  Когда масштаб 1000+ КДЛ / 1000+ работодателей — нужна ERM.
  Тогда White-label для МИС становится логичным.
```

**Предпочтение**: самостоятельность → Инвитро если надо → МИС когда масштаб.

## 3. Главные риски и как закрываются

| Риск | Ответ |
|---|---|
| "КДЛ сами это сделают" | У них конфликт интересов + legacy IT. У Инвитро дозаказ ручной. ML-триаж — другой уровень |
| Каннибализация чеков КДЛ | Мы ДОБАВЛЯЕМ, не заменяем. Контуры доступа + сегментация |
| Регуляторика / SaMD | Не ставим диагнозы, не заменяем врача. Работаем как CDSS / decision-support |
| "Это просто калькулятор" | Policy-движок + экономическая оптимизация маршрутов, не пороги |
| FP в "умном доборе" = "развели" | Высокий порог (P>0.6), мягкая формулировка, маленькая сумма (425р), скидка |
| Модель неточна на РФ данных | Валидация на реальных данных КДЛ-пилота. NHANES — старт, не финиш |
| Pre-analytical не продаваем | Не core, а потенциальный бонус-модуль. Custdev покажет |

## 4. Приоритеты запуска

```
СЕЙЧАС (6 недель):
  ├── B2C лендинг (CatBoost + gauge + SHAP + перцентили)
  └── КДЛ "умный добор" (pitch + пилот + первые SMS)
  → proof of concept, первые метрики, первая выручка

ЭТАП 2 (3-6 месяцев):
  ├── Масштабирование КДЛ (5-10 точек)
  ├── B2B работодатели (биохим-аудит через КДЛ/клинику)
  └── Рост B2C (OCR, ЛК, динамика, подписка)

ЭТАП 3 (6-12 месяцев):
  ├── Инвитро-партнёрство ИЛИ самостоятельный масштаб
  ├── Вторая вертикаль (преддиабет)
  └── Переговоры со страховыми (аудит)

МАСШТАБ (12+ месяцев):
  ├── МИС-интеграция (если нужна ERM)
  └── Страховые (длинный цикл, большие чеки)
```

### Стратегия независимости

```
ИДЕАЛ:    Свой B2C + прямые контракты с КДЛ + IP-лицензирование
ПРИЕМЛЕМ: Инвитро-интеграция (если их ИС подходит, не теряя IP)
ПОЗЖЕ:    МИС-модуль (когда 1000+ точек, нужна ERM)
КРАСИВО:  Самостоятельный бренд, не white-label чужого решения
```

## 5. Почему это стартап, а не идея

1. **Не конкурируем с КДЛ** — решаем конфликт интересов и неэффективность экосистемы
2. **Несколько БМ** (не одна):
   - B2C: freemium + подписка + CPA с КДЛ
   - B2B для КДЛ: % от инкрементальной маржи
   - B2B для корпоратов/ДМС: PEPY + доборы
   - B2B-аудит для страховых: SaaS + % от экономии
3. **Одна модель, разные обёртки** → экономия R&D
4. **Данные NHANES** → бесплатный, nationally representative датасет
5. **Первая выручка через 4-6 недель** (КДЛ-пилот + B2C)
6. **Литературная база** (MGH, CatBoost IDA, BMJ Open) → не фантазия, а validated approach

## 6. Одна модель, три обёртки

```
          NHANES (все фичи)
                |
        CatBoost (одна модель)
                |
    +-----------+-----------+
    |           |           |
  КДЛ         B2C        B2B/29н
  (лаб only)  (лаб+демо) (лаб+демо+физ)
  NaN для     NaN для    Полный набор
  физ.данных  физ.данных
    |           |           |
  AUC ~0.85   AUC ~0.85   AUC ~0.90
```

Треним на ВСЕХ фичах (включая ИМТ, АД). CatBoost нативно обрабатывает NaN.
Если канал не передаёт ИМТ/АД — модель работает, просто чуть менее точно.
