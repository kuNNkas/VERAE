# VISION: Независимый клинико-экономический Decision Engine

## Питч (одной фразой)

Мы делаем независимый AI-движок для лабораторной диагностики, который оптимизирует
назначения и доборы анализов, снижает операционные ошибки и перерасход, и повышает
клиническую ценность диагностики — через встраивание в МИС, корпоративные программы и ДМС.

## Что мы строим

Не "предсказание болезней по анализам".
Независимый клинико-экономический decision engine, который:

- Анализирует рутинные лабораторные данные + анкету
- Не ставит диагнозы, а:
  - Валидирует назначения
  - Делает триаж (кому и какие доборы реально нужны)
  - Снижает ошибки pre-analytical этапа
  - Оптимизирует расходы на диагностику
- Работает как встраиваемый мозг в МИС/ЛИС или как слой для страховых/работодателей

## Ключевая ценность

Делать лабораторную диагностику дешевле, безопаснее и осмысленнее для тех, кто платит
за систему (работодатель, страховая, клиника), не вступая в конфликт интересов с КДЛ.

## Почему мы, а не КДЛ / МИС сами

| Игрок | Почему не сделают сами |
|---|---|
| КДЛ (Инвитро, Гемотест) | Конфликт интересов: им выгодно делать больше тестов, не меньше. Legacy IT. |
| МИС-вендоры | Нет ML/R&D кадров. Они кодеры, не врачи-учёные. Им дешевле купить модуль. |
| Страховые | IT заточено под финансы, не под клинический ML. |
| Работодатели | Нет экспертизы в лабораторной медицине вообще. |

Мы встаём между интересами: КДЛ хочет больше тестов, страховая хочет меньше,
работодатель хочет здоровых сотрудников. Наш движок балансирует это.

## Реальная стратегия входа

```
ЕСТЬ:   Выходы на КДЛ и УКБ
ЕСТЬ:   ML/R&D-экспертиза (мы)
МОЖНО:  Запустить B2C самим
НЕТ:    Выходов на МИС/страховых (пока)
```

Порядок:
1. КДЛ/УКБ: pre-analytical валидатор -> кейсы + данные
2. B2C: "загрузи анализы" -> proof of demand + трафик
3. B2B: работодатели с 29н -> деньги
4. МИС/страховые -> масштаб (когда есть proof of concept)

## Ядро технологии (что у нас "не калькулятор")

### Risk & Policy Engine
- Не просто пороги (TSH<0.3), а выбор маршрута с учётом:
  - Клинического риска (красные флаги)
  - Вероятности полезности добора
  - Вероятности комплаенса пациента
  - Экономики канала (DTC vs ДМС vs корпоратив)
  - Операционных ограничений (реагенты, очереди, сезонность)

### Pre-analytical Validator
- Проверка корректности назначения и условий интерпретации:
  - Ферритин при воспалении -> рекомендовать CRP/sTfR
  - Бессмысленные комбинации анализов
  - Неподходящие пробирки/подготовка пациента

### Conversion Engine (этично)
- Кому предложить добор, когда, каким текстом, чтобы:
  - Не каннибализировать дорогие панели
  - Не создавать ощущение "развода"
  - Повышать конверсию только там, где есть клиническая польза

## Три режима работы (из литературы)

| Режим | Что делает | Источник |
|---|---|---|
| **Reflex** | CBC -> "нужен ферритин, добери" | McDermott & Baron, MGH (Smart Reflex) |
| **Low Info** | "Этот тест будет нормальным, не назначай" | Levick & Baron, MGH (Low Information Tests) |
| **Predict** | "Вот вероятность дефицита железа без теста" | MGH ferritin prediction model |

Все три = одна модель с разным порогом:
- P(ID) > 0.6 -> reflex (добери)
- P(ID) < 0.1 -> low info (не назначай)
- Между ними -> зона врачебного решения
